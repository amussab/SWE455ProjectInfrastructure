Executive Summary

The project implements a Secure File Validation Service. The business logic is intentionally focused: users upload files, the system checks whether the file type is safe and consistent with its content, then stores a result that users and admins can query.

The architecture satisfies the course requirement of at least two functional services and one data storage service. The two functional services are the Upload Service and Validator Service. The storage layer is DynamoDB for upload records and S3 for uploaded file objects. SQS decouples upload from validation so validation can run asynchronously.

Runtime Flow

  1. Client sends a multipart file request to API Gateway.
  2. API Gateway invokes the Upload Lambda.
  3. Upload Lambda stores the file in S3 and creates a PENDING record in DynamoDB.
  4. S3 emits an object-created event to SQS.
  5. SQS invokes the Validator Lambda.
  6. Validator Lambda reads the S3 object, validates the file, and updates DynamoDB.
  7. User/admin APIs read the latest status from DynamoDB.

Services

Upload Service

Repository: SWE455-upload-service

Cloud entrypoint: index.handler

Accepts authenticated upload requests, parses multipart content, writes file objects to S3, and creates initial DynamoDB records.

Validator Service

Repository: SWE455-validator-service

Cloud entrypoint: index.handler

Processes SQS messages from S3, checks file extension, size, MIME, magic bytes, executable spoofing, and CSV structure, then updates DynamoDB.

S3

Stores uploaded files and emits object-created events.

SQS

Buffers validation work and decouples upload latency from validation processing.

DynamoDB

Stores upload status, metadata, validation result, and rejection reason.

15-Factor Application Mapping

Factor How the Project Satisfies It
1. CodebaseSeparate GitHub repositories for infrastructure, upload service, and validator service.
2. DependenciesPython dependencies are declared in requirements.txt; validator uses standard library plus AWS runtime boto3.
3. ConfigRuntime configuration is passed through Lambda environment variables: bucket name, table name, API keys, size limit.
4. Backing ServicesS3, SQS, DynamoDB, Lambda, and API Gateway are attached resources managed through Terraform.
5. Build, Release, RunBuild scripts create zip artifacts; Terraform releases infrastructure; Lambda runs immutable uploaded artifacts.
6. ProcessesLambdas are stateless processes. Durable state lives in S3 and DynamoDB.
7. Port BindingAPI Gateway exposes HTTP routes. Lambda code does not require manually managed ports.
8. ConcurrencyLambda and SQS scale concurrent validation work automatically.
9. DisposabilityShort-lived Lambda invocations start quickly and fail messages back to SQS for retry.
10. Dev/Prod ParitySame Python service logic and zip artifacts are used for repeatable deployment.
11. LogsServices log to stdout/stderr, collected by CloudWatch.
12. Admin ProcessesTerraform scripts and artifact build scripts are used for setup, teardown, and deployment.
13. API FirstREST endpoints and OpenAPI documentation are provided in the upload-service repository.
14. TelemetryHealth, readiness, metrics endpoints, Lambda logs, and AWS metrics support observability.
15. Authentication & AuthorizationUser endpoints require x-api-key; admin endpoints require x-admin-key.

Current Service Contract

Component Contract Important Runtime Config
Upload Lambda Receives HTTP requests from API Gateway and writes the file to S3 plus a PENDING record to DynamoDB. API_KEY, ADMIN_API_KEY, UPLOAD_BUCKET_NAME, UPLOADS_TABLE_NAME, MAX_UPLOAD_BYTES.
S3 Upload Bucket Stores raw file objects and emits object-created events to SQS. Object metadata: upload-id, user-id, original-filename.
SQS Queue Buffers validation work and invokes the validator Lambda through event source mapping. Queue policy allows S3 to send messages from the upload bucket.
Validator Lambda Reads S3 object, validates content, and updates the matching DynamoDB record. UPLOADS_TABLE_NAME, MAX_FILE_SIZE_BYTES, LOG_LEVEL.
DynamoDB Single source of truth for user history, admin upload list, alerts, and validation status. Table: file-validation-results; key: file_id.

Why the Architecture Uses SQS

SQS is used to decouple the upload request from validation. The upload API can return quickly after the file is stored and a pending record is created. The validator can process files independently, retry failed messages, and scale based on queue depth. This directly supports concurrency, disposability, and backing-service separation in the 15-Factor methodology.

IAM Decisions

The current Terraform uses AWS-managed policies for S3, SQS, DynamoDB, and Lambda execution. This keeps the course demo simple and reproducible. A production hardening step would replace those broad policies with custom least-privilege policies scoped to the exact upload bucket, queue, and DynamoDB table.

Scaling and Failure Handling

Concern Design Decision
Upload traffic spikes API Gateway and Lambda scale automatically for incoming HTTP requests.
Validation load SQS buffers work and validator Lambda scales as messages arrive.
Validation failure Unexpected processing errors raise from the validator so SQS can retry.
Rejected files Rejected and suspicious files are stored as normal validation outcomes, not treated as Lambda crashes.
Logs and telemetry Lambda writes to stdout/stderr, and CloudWatch stores runtime logs and metrics.
